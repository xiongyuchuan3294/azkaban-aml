# 反洗钱大数据自动化测试框架方案（3-Skill 分析主链版）

## 1. 目标定位

本方案服务日常提测、联调验证、缺陷复测和高频改动点稳定性检查。  
核心输入保持双源：

- 需求文档
- 项目代码 `git diff`/commit

核心原则：先把“需求理解 + 代码变更 + 影响闭包”三段式打通，再逐步扩展到全链路自动化执行。

## 2. 建设策略

采用“3 个核心分析 Skill + 执行验证 Skill 链”的分层建设方式：

1. 先产出结构化需求和结构化变更。  
2. 再做合并分析，直接输出血缘关系、最小闭包 DAG 和测试节点范围。  
3. 下游用例、造数、执行、校验模块消费统一的影响分析报告。

## 3. 强制输入与门禁

每个需求必须提供：

| 工件 | 必填字段 | 用途 |
|---|---|---|
| 需求文档 | `requirement_id`、业务规则、边界、验收标准 | 生成测试目标与断言口径 |
| Git Diff/Commit | commit范围、变更文件、关键逻辑改动 | 识别真实改动点 |
| 项目路径 | repo路径（如 `/Users/xiongyuc/workspace/azkaban-aml`） | 提取节点依赖和表血缘 |

门禁规则：

- `Gate-1`：需求文档可解析（产出 `requirement_report`）。
- `Gate-2`：git diff 可解析（产出 `commit_report`）。
- `Gate-3`：需求点与改动文件可映射（映射覆盖率达标）。
- `Gate-4`：可形成最小可测闭包（`must_run_nodes` 非空）。

未通过门禁的需求不执行下游，仅输出阻断原因和补充建议。

## 4. Skill 清单（按最新方案）

### Skill-1：`requirement-parser`
- 职责：解析需求文档，抽取功能点、输入输出、表映射、主逻辑、候选节点。
- 输入：需求文档（md/text/json）。
- 输出：`requirement_report.md`（可选 `requirement_report.json`）。
- 验收：功能点和关键规则可稳定抽取，缺失信息有澄清提示。

### Skill-2：`diff-parser`
- 职责：解析 `git diff`/commit 区间，提取改动文件与改动摘要。
- 输入：repo + 过滤条件（关键词/作者/日期）。
- 输出：`commit_report.json`。
- 验收：变更文件和摘要可稳定生成，结果精简可读。

### Skill-3：`requirement-impact-analysis`
- 职责：将需求报告与代码变更合并分析，输出：
  - 需求点 <-> 变更文件映射
  - 血缘扩展
  - 最小闭包 DAG
  - 测试节点与表范围
- 输入：`requirement_report.(md|json)` + `commit_report.json` + `repo`
- 输出：`impact_analysis_report.json`（可选 `.md`）
- 验收：可稳定产出 `must_run_nodes`、`validation_nodes`、`node_test_scope`。

### Skill-4：`test-scope-planner`（可选）
- 职责：基于 `impact_analysis_report` 做主题级/风险级测试范围裁剪。
- 输入：`impact_analysis_report.json`
- 输出：`test_scope.yaml`

### Skill-5：`test-case-planner`
- 职责：生成正例/边界/异常用例及断言。
- 输入：`requirement_report` + `impact_analysis_report`（或 `test_scope.yaml`）
- 输出：`test_cases.yaml`

### Skill-6：`test-data-generator`
- 职责：按场景自动造数、隔离、回收。
- 输入：`test_cases.yaml`
- 输出：`data_plan.yaml`

### Skill-7：`execution-orchestrator`
- 职责：统一编排 Azkaban/Hive/MySQL 执行。
- 输入：`impact_analysis_report.json` + `data_plan.yaml` + `test_cases.yaml`
- 输出：`execution_log.json`

### Skill-8：`result-validator`
- 职责：执行完整性/准确性/时效性校验。
- 输入：执行结果 + 断言规则
- 输出：`assertion_result.json`

### Skill-9：`report-archiver`
- 职责：生成追溯矩阵与测试报告并归档证据。
- 输入：全链路中间产物
- 输出：归档包、需求级测试报告

## 5. 运行模式

### 5.1 日常测试模式（默认）

主链：
`requirement-parser -> diff-parser -> requirement-impact-analysis -> test-case-planner -> test-data-generator -> execution-orchestrator -> result-validator -> report-archiver`

特点：
- 快速反馈
- 小范围定向验证
- 支持联调与缺陷复测

### 5.2 自动化回归模式（按需）

主链：
`requirement-parser -> diff-parser -> requirement-impact-analysis -> test-scope-planner -> test-case-planner -> test-data-generator -> execution-orchestrator -> result-validator -> report-archiver`

特点：
- 基于影响范围做分层覆盖
- 适合发布前集中回归

## 6. 模块接口契约

统一约束三类接口：

- 输入：上游标准产物文件
- 输出：下游标准产物文件
- 元数据：`requirement_id`、`run_id`、`status`、`error_code`

建议目录：

- `inputs/`
- `outputs/`
- `logs/`

## 7. 关键产物与字段

核心产物：

- `requirement_report.(md|json)`
- `commit_report.json`
- `impact_analysis_report.(json|md)`

推荐追溯字段：

- `requirement_id`
- `requirement_item_id`
- `git_commit_range`
- `changed_file`
- `mapped_confidence`
- `must_run_nodes`
- `validation_nodes`
- `dependency_tables`
- `result_tables`
- `test_case_id`
- `assertion_id`
- `run_id`
- `result_status`

## 8. 实施路线（按最新三段式）

### 阶段1（1-2 周）：实现分析主链
- `requirement-parser`
- `diff-parser`
- `requirement-impact-analysis`

验收：
- 任一需求可稳定产出映射结果、最小闭包 DAG、节点与表测试范围。

### 阶段2（2-3 周）：实现用例与造数链
- `test-scope-planner`（可选）
- `test-case-planner`
- `test-data-generator`

验收：
- 需求点可自动生成用例与造数计划。
- 造数可重复执行且结果一致。

### 阶段3（1-2 周）：实现执行与校验链
- `execution-orchestrator`
- `result-validator`

验收：
- 可自动执行并自动产出断言结果。
- 失败可定位，支持重试。

### 阶段4（1 周）：实现报告与全链路串联
- `report-archiver`
- 全链路联调与稳定化

验收：
- 端到端链路跑通。
- 连续执行稳定，支持日常常态化使用。

## 9. 与现有环境结合

基于现有 Azkaban + Hive + MySQL：

- 复用现有执行能力，不重复造平台。
- 在 `requirement-impact-analysis` 中自动读取 `jobs/*.job` 与 `sql/*.sql` 生成节点/表血缘。
- 如后续接入企业级血缘平台，可替换为平台血缘输入，保持产物结构不变。

## 10. 核心指标

- 日常需求自动化覆盖率
- 单需求平均测试准备时长
- 首次自动化反馈时长
- 需求点-改动点映射覆盖率
- 最小闭包节点命中率
- 造数重跑一致性
- 缺陷复测自动化命中率

## 11. 简化落地建议

- 首期优先覆盖高频 AML 需求类型。
- 首期只启用高价值断言，不一次做全规则引擎。
- 报告先满足“可追溯、可复盘”，后续再做管理看板。

## 12. 预期收益

- 3 个核心分析 Skill 直接闭环，减少步骤割裂与维护成本。
- 需求、变更、血缘、测试范围在同一报告汇总，沟通成本更低。
- 从日常测试切入，可平滑演进到发布级全链路自动化。
